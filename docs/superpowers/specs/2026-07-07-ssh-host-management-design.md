# SSH Host Management Design

Date: 2026-07-07
Status: Approved for implementation

## Summary

Project Pilot will add first-class SSH Host records so several SSH projects can share one connection definition. Editing a Host, including replacing its IP address, will immediately affect every linked project. The Manager webview will provide Host CRUD, while the native VS Code OUTLINE TreeView will upgrade its existing `By Target` mode to a real `By Host` mode.

This feature also includes fixes for bugs found in the directly affected SSH path, Remote-SSH authority, Host grouping, and connection-test flows. Unrelated refactoring and unrelated bug fixes are out of scope.

## Goals

- Store an SSH connection once and reference it from several projects.
- Manage Host display name, hostname or IP address, optional username, and optional port.
- Change a Host address without manually editing every project.
- Automatically migrate safely parseable legacy SSH projects.
- Manage and browse projects by Host in the native OUTLINE TreeView.
- Preserve existing project configurations when a legacy SSH path cannot be migrated safely.
- Preserve the current uncommitted SSH authority and username-normalization fixes and cover them with regression tests.

## Non-goals

- Editing the user's `~/.ssh/config`.
- Managing private keys, passwords, secrets, `ProxyJump`, or arbitrary SSH options.
- Replacing VS Code Remote-SSH configuration.
- Automatically merging an SSH config alias with an IP address merely because they currently resolve to the same machine.
- Fixing bugs outside SSH Host management, SSH path resolution, and OUTLINE Host grouping.

## User-approved decisions

- A Host manages hostname or IP, username, and port.
- Existing SSH projects are migrated automatically when they can be parsed safely.
- A Host referenced by projects cannot be deleted.
- The existing native OUTLINE `By Target` mode becomes `By Host`.
- Hosts are first-class stored entities; this is not implemented as string replacement.
- Favorites and Recent remain flat and are not nested under Hosts.

## Data model

The stored state advances to schema version 2:

```ts
interface SshHost {
  id: string;
  name: string;
  hostname: string;
  username?: string;
  port?: number;
}

interface State {
  schemaVersion: 2;
  sshHosts: SshHost[];
  projects: ProjectItem[];
  uiSettings?: UISettings;
}
```

Managed SSH projects add two canonical fields:

```ts
interface ManagedSshProjectFields {
  sshHostId: string;
  remotePath: string;
}
```

For a managed SSH project, `sshHostId` and `remotePath` are the source of truth. The existing `path` field remains a derived compatibility snapshot. With no explicit custom port, it stays in the readable `user@host:/repo` form. With an explicit custom port, it is a structured `vscode-remote://ssh-remote+...` URI so older Project Pilot versions retain the port when opening the project. Runtime code must not use the snapshot when managed fields are present. It is regenerated before persistence and export.

Legacy SSH projects without `sshHostId` continue to use `path` as their source of truth until migration succeeds.

### Host identity and validation

- `id` is immutable and generated by Project Pilot.
- `name` is a user-facing alias and must be unique case-insensitively.
- `hostname` is required after trimming and may be an SSH config alias, DNS name, IPv4 address, or IPv6 address.
- `username` is optional; an omitted value allows OpenSSH configuration and local defaults to apply.
- `port` is optional; when present it must be an integer from 1 through 65535.
- Duplicate normalized connection tuples `(username, hostname, port mode)` are rejected. An omitted port means "defer to SSH configuration or the SSH default" and is distinct from explicitly storing port 22.

## Central SSH project resolver

One extension-host module will resolve every SSH project operation:

```ts
resolveSshProject(project, sshHosts) -> {
  host,
  remotePath,
  displayPath,
  rawSshPath,
  remoteUri
}
```

The resolver is used for opening, display labels, copying paths, connection tests, name suggestions, OUTLINE grouping, import/export materialization, and webview state. This prevents the Manager, OUTLINE, and commands from assembling SSH addresses differently.

When username or port is explicit, `remoteUri` uses the structured hexadecimal Remote-SSH authority containing `hostName`, optional `user`, and optional `port`. Plain safe authorities remain readable. Parsing must accept both forms. This extends the current authority-normalization work instead of replacing it.

Remote paths continue to support POSIX paths, Windows drive-letter paths, `.code-workspace` files, raw SSH strings, and `vscode-remote://ssh-remote+...` URIs.

## Legacy migration

The same normalization and migration pipeline runs for every incoming state: store initialization, watched-file reload, manual reload, import, and backup restore. It finishes before views receive the new state.

1. Ensure `projects`, `sshHosts`, and `uiSettings` exist.
2. Map stored OUTLINE mode `target` to `host`.
3. Leave local and local workspace projects unchanged.
4. For each SSH project without managed fields, parse its raw SSH path or Remote-SSH URI.
5. Normalize username, hostname, optional explicit port, and remote path without DNS lookup or other network access.
6. Reuse an existing Host with the same normalized connection tuple, or create one.
7. Store the Host ID and remote path, then regenerate the compatibility `path` snapshot.
8. If parsing is unsafe or incomplete, keep the legacy project unchanged and record a non-blocking migration warning for the UI.
9. Save the upgraded state once after the whole migration succeeds.

Migration is idempotent. Running it again must not create new Hosts or modify already managed projects. Host names default to the normalized hostname; case-insensitive collisions receive a deterministic numeric suffix.

The extension must keep the original in-memory state until validation and migration complete. A failed migration leaves the previous file unchanged and reports the error.

## Store operations

`ConfigStore` becomes the single mutation boundary for Hosts and references:

- `addSshHost(host)` validates and inserts a Host.
- `updateSshHost(host)` validates and updates one Host.
- `deleteSshHost(id)` rejects deletion while any project references the Host.
- `migrateSshHostProjects(sourceId, targetId, projectIds?)` reassigns all or selected projects while preserving their remote paths.
- `upsertProject(project)` validates that managed SSH projects reference an existing Host.
- `resolveProject(project)` returns a complete runtime SSH target.

Each operation constructs and validates the next state, updates all derived compatibility paths, and performs one save. If saving fails, the previous in-memory state is restored and views remain on the last persisted state.

Editing a Host to a tuple already owned by another Host is rejected with a prompt to migrate projects to the existing Host. Silent merge is not allowed.

## Manager webview

The Manager toolbar gains an `SSH Hosts` action that opens a Host management panel. The panel shows:

- Host name.
- Effective `user@hostname:port` label.
- Number of linked projects.
- Add, edit, test, migrate, and delete actions as allowed by state.

The SSH project editor changes from one full SSH path field to:

- Host selector.
- `New Host` shortcut.
- Remote path field.
- Existing type, metadata, group, tags, icon, and color fields.

Legacy unparseable projects continue to show the existing full-path editor plus a migration warning. They can be converted manually by choosing a Host and confirming the remote path.

Webview messages are explicit operations rather than generic state replacement:

- `addSshHost`
- `updateSshHost`
- `deleteSshHost`
- `testSshHost`
- `migrateSshHostProjects`

The extension host validates every payload; the webview's validation is only for immediate feedback.

## Native OUTLINE TreeView

The current `OutlineMode` value `target` is renamed to `host`, with persisted `target` values migrated automatically. The mode order is:

1. `By Group`
2. `By Host`
3. `By Type`
4. `Flat`

Favorites and Recent preserve their current flat structure. In `By Host`, the `All Projects` section contains:

- One expandable node for each referenced SSH Host.
- A display-only `Local` pseudo-host for local folders and local workspace files.
- Linked project children under each node.

Host nodes show their name and project count. Their tooltip includes hostname, username, port, and the linked project list. Their context menu provides:

- Edit Host.
- Test Connection.
- Migrate Projects.
- Delete Host only when its reference count is zero.

Project nodes retain their existing open, copy, favorite, move, test, and delete actions.

The OUTLINE edit and migration commands open focused VS Code QuickPick/InputBox workflows or the Manager Host panel. No second TreeView is added.

## Connection testing

The existing connection test must no longer report success after checking only string syntax.

Testing a Host performs two bounded stages:

1. Run `ssh -G` with the Host's explicit username and port to show the effective hostname, username, and port.
2. Run a non-interactive OpenSSH probe with `BatchMode=yes` and a short connection timeout.

The UI distinguishes executable-not-found, invalid Host, DNS failure, timeout, host-key failure, authentication failure, and remote-command failure. Password-only Hosts may fail the non-interactive authentication stage; the message must explain that limitation without treating configuration parsing as a successful connection.

Connection testing never writes SSH configuration and never requests or stores credentials.

## Error handling

- Invalid Host fields are rejected before state mutation.
- Missing Host references are treated as recoverable configuration errors. The project remains visible but cannot open until repaired.
- Deleting a referenced Host is blocked and lists the referencing projects.
- Duplicate Host tuples are blocked and offer migration to the existing Host.
- Failed writes restore the previous state and show an error.
- Failed migration leaves the legacy project intact.
- Webview messages with unknown Host or project IDs return an error result and do not mutate state.
- View refresh happens only after a successful store operation.

## Import, export, and backups

- Import accepts schema version 2, the current `{ projects: [...] }` format, bare project arrays, and supported legacy folder structures.
- Legacy imports run the same idempotent migration as startup.
- Schema version 2 exports include `sshHosts`, managed project references, `remotePath`, and regenerated complete `path` snapshots.
- Import validation checks Host IDs, duplicate Host names and tuples, project references, ports, and SSH project paths.
- Existing backup behavior remains in place; imports still create backups according to current settings.

## Testing strategy

The repository has lightweight Node assertion tests in the current worktree. The implementation will extend that pattern rather than introduce a new test framework.

Automated coverage includes:

- Host validation and normalized tuple uniqueness.
- Legacy migration, deduplication, idempotence, and unparseable fallback.
- Managed path resolution and compatibility snapshot regeneration.
- POSIX paths, Windows drive-letter paths, SSH workspaces, IPv4, IPv6, usernames, and custom ports.
- Plain, percent-encoded, and structured hexadecimal Remote-SSH authorities.
- Existing local-username and structured-authority regressions.
- Host update, deletion protection, project migration, and failed-save rollback.
- OUTLINE grouping and persisted `target` to `host` mode migration through pure helpers.
- Connection probe argument construction and error classification with mocked processes.
- Manager message routing for Host operations where it can be isolated from VS Code APIs.

Verification requires:

```bash
npm run test:sshPath
npm run test:sshResolve
npm run test:sshHosts
npm run build
```

Manual Extension Host verification covers:

1. Automatic migration from a copy of a legacy configuration.
2. Host add, edit, duplicate rejection, connection test, migration, and deletion protection.
3. SSH project add and edit using Host plus remote path.
4. OUTLINE cycling and `By Host` grouping in the exact native TreeView shown by the user.
5. Updating one Host IP and opening several linked folders and workspace files.
6. Import/export round trips for both new and legacy formats.
7. Local, Linux SSH, Windows SSH, and `.code-workspace` path cases.

## Rollout and compatibility

- No destructive migration is performed for an unparseable project.
- Existing SSH authority fixes in the dirty worktree are treated as pre-existing user work and preserved.
- New code must route through the central resolver while legacy fallback remains until all projects can be migrated.
- The generated `.superpowers/` visual-companion directory is not part of the product change and will not be committed.
